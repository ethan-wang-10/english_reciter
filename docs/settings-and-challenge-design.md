# 设计：打卡门槛、用户设置与月度 PK 挑战

本文档描述产品行为、数据与接口方向，便于评审后再实现。实现时需与现有 `gamification.py`（`user_data_simple/<user>/gamification.json`）及 `simple_web_app.py` 对齐。

---

## 1. 打卡规则变更：至少复习 5 个单词

### 1.1 现状（摘要）

- 每次「答对」会调用 `award_correct_answer`，在 `streak_correct_by_day[今日]` 上累加答对次数，并调用 `_update_streak`。
- `_update_streak` 判定「当日已打卡」：当日 `streak_correct_by_day` 有值 **或** `daily_xp` 当日有值（即至少答对过 1 次且可能产生 XP）。

### 1.2 目标

- **当日计为「有效打卡」**：当日累计 **答对次数 ≥ 5**（建议：主轮 + 错题巩固 + 加练均计入，与现有一致来源）。
- **连续打卡（streak）**：仅在「当日为有效打卡日」时，才按现有逻辑更新连续天数（与昨日是否连续、是否断档等规则保持一致）。

### 1.3 逻辑要点

1. **判定时机**：在每次答对后更新 `streak_correct_by_day` 之后，用**当日累计答对次数**与阈值 `5` 比较；仅当 `>= 5` 时认为本日满足打卡条件，再执行 `_update_streak`（或把 `_update_streak` 拆成「是否满足打卡条件」+「更新连续天数」两步）。
2. **XP 与打卡解耦**：打卡门槛只依赖「答对次数」，不因 XP 被软上限压成 0 而失效；但若当日从未达 5 次答对，则**不**更新 streak（即使有一次答对产生了 XP）。
3. **文案**：
   - 导航/展示中的「🔥 连续 N 天」仍表示连续**有效打卡**天数。
   - 若需区分「今日已答对 0～4 词」的提示，可在前端或接口中增加 `today_correct_count` / `check_in_done_today` 供 UI 展示。

### 1.4 迁移与兼容

- 老数据已有 `streak_correct_by_day` 历史；**规则从上线日起**按新阈值执行即可。
- 历史 streak 不自动重算；如需「公平性」可另议（一般不做回溯）。

### 1.5 待产品确认（可选）

- 「5 个」是否包含**同一单词多次答对**（当前为每次答对 +1）：默认 **包含**，与「复习次数」一致。
- 是否仅统计「主轮」答对：默认 **否**，与现有一致（主轮 + 错题 + 加练均计入）。

## 2. 用户设置：入口、头像、月度打卡目标

### 2.1 入口与信息架构

- **入口**：顶栏点击 **用户名**（`#username-display`）→ 打开 **设置**（侧滑抽屉 / 全屏子页 / 模态层三选一；推荐 **抽屉或独立 section**，与现有路由 `showSection` 风格一致）。
- **设置内区块**：
  1. 头像（上传 / 裁剪 / 删除恢复默认）
  2. 本月打卡天数目标（数字输入或滑块）
  3. 本月进度（见 2.4）

### 2.2 头像

- **存储**：`user_data_simple/<username>/avatar.<ext>`（或统一转 WebP），**不**进 Git（已在 `user_data_simple/*` ignore 范围内）。
- **限制**：类型 `image/jpeg`、`image/png`、`image/webp`；大小上限如 2MB；服务端缩放至如 256×256。
- **API 草案**：
  - `POST /api/user/avatar`：`multipart/form-data`，返回 `{ "url": "/api/user/avatar?user=..." }` 或带 token 的静态路径。
  - `DELETE /api/user/avatar`：删除文件，前端用默认头像。
- **默认头像**：首字母或通用 SVG，与排行榜展示一致。

### 2.3 本月打卡天数目标

- **语义**：用户设定「本月希望完成有效打卡的天数」目标（例如 20 天）。
- **范围**：建议 1～31 或 1～当月天数，超出则校验失败。
- **存储**：`gamification.json` 增加字段，例如  
  `monthly_checkin_goal: { "YYYY-MM": 20 }` 或仅保存「当前月」一条 `current_month_goal` + 月份键，避免无限增长。

### 2.4 当月进度（用户可见）

- **定义**：当月「有效打卡」的天数 = 当月日期集合中，满足 **当日答对 ≥ 5** 的天数（与第一节一致）。
- **展示**：
  - 进度条：`本月已打卡天数 / 目标天数`（目标可为空，则只显示「已打卡 X 天」）。
  - 可选：简易日历热力图（仅当月），与 `streak_correct_by_day` 按日汇总兼容。

### 2.5 API 草案（设置）

- `GET /api/user/settings`：返回 `{ avatar_url, monthly_goal, month_progress, month_key }` 等。
- `PATCH /api/user/settings`：`{ monthly_goal?: number }`。
- 头像单独 `POST/DELETE` 如上。

## 3. 用户间挑战：PK 每月打卡天数

### 3.1 目标与范围

- **PK 指标**：**自然月内**双方各自「有效打卡天数」（与 1、2 节定义一致）。
- **周期**：每月 1 日 0 点（服务器时区，建议统一 `Asia/Shanghai`）起算，月末结算；**不**跨月累计到同一局 PK。

### 3.2 流程（建议 MVP）

1. **发起**：用户 A 选择用户 B（需已知用户名或从排行榜/好友列表选），发起「本月打卡挑战」。
2. **接受/拒绝**：B 收到待处理请求（或站内列表），接受后双方在本月绑定同一 **challenge_id**。
3. **进行中**：双方各看自己的当月进度；**对比页**展示双方当月有效打卡天数（不展示对方每日详情，除非产品要求）。
4. **结算**：下月 1 日或月末 24 点后生成结果：胜 / 负 / 平（天数相同）；可选记录历史。

### 3.3 隐私与同意

- 仅 **互为挑战关系** 或 **双方均开启挑战可见** 时，才能看到对方当月打卡天数。
- 与现有 `leaderboard_opt_in` 可独立：**挑战 PK 单独 opt-in**（例如 `challenge_opt_in`），避免强行曝光。

### 3.4 数据模型（草案）

- 新文件：`user_data_simple/_challenges/challenges.json`（或分表），结构示例：

```json
{
  "challenges": [
    {
      "id": "uuid",
      "month": "2025-03",
      "user_a": "alice",
      "user_b": "bob",
      "status": "pending|active|declined|completed",
      "created_at": "ISO8601",
      "accepted_at": null
    }
  ]
}
```

- 结算时写入 `result: { winner: "alice"|"bob"|"tie", a_days: 12, b_days: 10 }`。

### 3.5 API 草案（挑战）

- `POST /api/challenges`：`{ "target_username": "bob", "month": "2025-03" }` 可选月份默认当月。
- `GET /api/challenges`：我的列表（待处理、进行中、已结束）。
- `POST /api/challenges/<id>/accept` | `decline`。
- `GET /api/challenges/<id>`：详情（含双方当月天数，权限校验）。

### 3.6 风险与边界

- **刷量**：打卡已依赖 ≥5 答对，降低作弊动机；仍依赖服务端计数。
- **时区**：全站统一一个时区，避免「跨日」争议。
- **性能**：排行榜 + 挑战列表需控制扫描用户数（已有 `users.json` 可枚举）。

## 7. 单人 PK：赌注 XP（胜者从败者处获得）

在 **第 3 节**「按月打卡天数 PK」基础上，增加 **可选项：赌注**。

### 7.1 规则摘要

- 双方在 **接受挑战时** 约定同一档 **赌注 XP**（见第 9 节推荐值），并从各自 `total_xp` 中 **冻结/扣除** 等额 XP 进入本局 **托管**（实现上可记 `escrow` 或先扣再奖，见 7.4）。
- **月末结算**：打卡天数多者胜 → **胜者获得 `赌注` XP**（来源为败者已托管部分）；败者失去该 XP。
- **平局**（当月有效打卡天数相同）：**全额退回**双方托管 XP，无人得失。

### 7.2 前置条件

- 双方 `total_xp >= 赌注` 方可接受；否则拒绝并提示「积分不足」。
- 同一自然月、同一对手是否允许多局：建议 **每对每月最多 1 局带赌注 PK**（避免重复抽血）。

### 7.3 与无赌注模式的关系

- **无赌注**：仅比较天数与展示胜负，**不**转移 XP（与旧版第 3 节一致）。
- 发起时必选：`wager_xp: 0 | 50 | 100 | 200`（具体档位见第 9 节）。

### 7.4 实现要点（账务）

- **推荐**：接受时双方各执行 `total_xp -= wager`（或写入 `challenge_escrow[challenge_id][user] = wager` 且展示「可用 XP」时减去冻结额），结算时 `winner.total_xp += 2 * wager` 若败者全额付出；或 `winner += wager`，**不**重复加系统币（奖池来自败者已扣）。
- 等价写法：败者已扣 `w`，胜者获得 `w`（净赚 `w`），败者净亏 `w`；系统无增发 XP。
- **审计**：`gamification.json` 或独立 `xp_ledger.json` 记录挑战 id、月份、双方 delta，便于对账与客服。

## 8. 月度群体打卡挑战（全站奖池）

### 8.1 规则摘要

- **每月第 1～5 天**（自然日，服务器时区）：用户可选择 **是否参加本月群体挑战**。
- 若参加：当场扣除 **入场费 XP（固定 150）** 进入 **当月奖池**；不参加则本月不参与瓜分。
- **次月第 1 天**（或当月最后一日 24:01）结算：统计参与者在本月 **有效打卡天数**（与第 1 节一致）；**打卡天数并列最高**的所有参与者 **平分当月奖池**（向下取整，余数作废或给系统舍入规则写死）。
- **未参与奖池者**：不参与排名与分红。

### 8.2 边界

- 若 **无人参加** 或 **奖池为 0**：不结算或原路退回（产品二选一；建议 **不退** 因无人参加时池为 0）。
- 若 **仅 1 人参加**：该人是否独吞全池？建议 **仍按「有效打卡天数最高」**；仅一人则其天数自动为最高 → **独吞奖池**（或产品要求最低参与人数 ≥2 才开奖）。
- **平局**：多名「最高天数」用户平分（与单人 PK 平局不同：这里是 **共享奖池**）。

### 8.3 数据与 API（草案）

- 全局：`user_data_simple/_challenges/monthly_pool.json`（示例）

```json
{
  "2025-03": {
    "pool_xp": 4500,
    "participants": ["u1", "u2", "..."],
    "status": "open|locked|settled",
    "settled_at": null
  }
}
```

- `POST /api/monthly-pool/join`：仅在当月 1～5 日可调用；扣费并入池。
- `GET /api/monthly-pool/current`：展示当前月奖池总额、已参与人数、自己是否已参与、截止日。

### 8.4 与单人 PK 的关系

- 群体挑战与 **1v1 赌注** 独立；同一用户可同时参与（注意 **总 XP 足够** 支付入场费 + 可能的多局托管）。

## 9. XP 赌注与入场费：数值规划（推荐）

以下基于当前代码中的 XP 产出（见 `gamification.py`）：

| 概念 | 约略量级 |
|------|-----------|
| 单次主轮答对基础 | `XP_PLAN_CORRECT = 10` |
| 含进度步进 / 掌握等单次上限 | 常见单题约 10～55+ |
| 每日 XP 软上限 | `DAILY_XP_SOFT_CAP = 300`，超出后减半 |

### 9.1 量级直觉

- **约 100 XP**：大致相当于 **正常复习 10 次左右答对**（未触顶），或 **半天～一天** 中等强度；作为 **1v1 赌注** 有痛感但一般不会「一天清零」。
- **150 XP（群体固定入场费）**：大致相当于 **触顶日（300）的一半**，或 **约 15 次主轮答对**；略低于旧案 200，降低参与门槛，奖池仍随人数放大。

### 9.2 推荐默认档（可配置常量）

| 用途 | 推荐默认值 | 可选档位 | 说明 |
|------|------------|----------|------|
| **1v1 赌注** | **100 XP** | **50 / 100 / 200** | 默认 100；新手可选 50；高风险 200。平局全额退回。 |
| **群体入场费** | **150 XP（固定）** | 仅一档 | 全站统一，不参与则无其它档位；便于规则简单、奖池可预期。 |

### 9.3 约束建议

- **最低总 XP**：参与任一带扣款玩法前，`total_xp >= 赌注或入场费 + 安全余量`（如 ≥50），避免扣成负数；若采用「冻结」，展示「可用 XP = total − 冻结」。
- **上限**：单次赌注不超过例如 **500 XP**，防止大 R 碾压与纠纷；群体入场费单档固定，**不按比例财富**。
- **经济平衡**：群体奖池为 **参与人数 × 150**；若 20 人参与则池 **3000 XP**；独吞或少数并列高分时仍有明确激励。

### 9.4 与等级公式的关系

- `level_from_xp` 使用 `sqrt(total_xp/100)` 量级；**输掉 100～200 XP** 通常 **不会掉级**（除非临界）；需在文案中说明「输掉的是已积累积分，非等级直接降级」以免误解（若等级由 total_xp 直接算，输 XP 可能掉级——产品可接受或改为「展示等级只升不降」另议）。
